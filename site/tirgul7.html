מהו Signal?

מכניזם שבעזרתו תהליכים מיודעים על מאורעות במערכת. איתות הוא מאורע אסינכרוני — תהליך צריך להיות מוכן לקבלו בכל שלב של ריצתו. הסיגנל לא חוסם את התהליך — הוא קורא במקביל מאחורי הקלעים עד שהמקבל תופס אותו.

מי יכול לאותת? מערכת ההפעלה, תהליך אחר, או התהליך עצמו. למה לאותת?

  • שינוי סטטוס תהליך — SIGKILL (סיום), SIGSTOP (עצירה), SIGCONT (המשך).
  • יידוע על בעיה — SIGSEGV (חריגה בזיכרון), SIGFPE (שגיאה אריתמטית, כמו חלוקה באפס).
  • תקשורת בין תהליכים — SIGUSR1, SIGUSR2.

בלינוקס סה"כ 64 סיגנלים. דומה ל-enum: ניתן לקרוא להם בשם, ולכל אחד גם value מספרי (למשל SIGFPE = 8).

סוגי פעולות (Action)

Term

סיום התהליך.

Ign

התעלמות מהאיתות.

Core

יצירת core dump file וסיום התהליך.

Stop

עצירת התהליך.

Cont

המשכת התהליך אם היה עצור.

כשמקבלים איתות, יש 3 אפשרויות:

  1. פעולת ברירת המחדל — SIG_DFL.
  2. התעלמות — חוץ משני איתותים שאסור להתעלם מהם: SIGKILL ו-SIGSTOP.
  3. כתיבת פונקציה שמטפלת בו.

signal() — הגדרת פונקציית טיפול

#include <signal.h> typedef void (*sighandler_t)(int); sighandler_t signal(int signum, sighandler_t handler);

פונקציית הטיפול היא void, מקבלת את מספר הסיגנל (int). signal מחזירה מצביע לפונקציה הקודמת שהייתה רשומה. כשקוראים ל-signal עם אותו signum אך handler חדש — המערכת מחליפה את הרישום הישן. אם יש שגיאה → SIG_ERR.

handler_demo.c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void on_sigint(int sig) {
    printf("\nCaught signal %d (SIGINT) — but I won't quit!\n", sig);
}

int main(void) {
    signal(SIGINT, on_sigint);   /* רושמים טיפול ל-Ctrl+C */
    while (1) {
        printf("working...\n");
        sleep(1);
    }
    return 0;
}

הסבר: בדרך כלל Ctrl+C שולח SIGINT שמסיים את התוכנית. כאן רשמנו handler משלנו, כך שבמקום לצאת — תודפס הודעה. הקלט ל-handler הוא מספר הסיגנל (2). שימו לב ש-SIGKILL לעולם לא ניתן ללכידה כזו.

הפונקציה kill

#include <sys/types.h> #include <signal.h> int kill(pid_t pid, int sig);

שולחת איתות לתהליך / קבוצת תהליכים. הצלחה → 0, כישלון → −1.

  • pid > 0 — שולח לתהליך שזה ה-PID שלו.
  • pid == 0 — שולח לכל התהליכים שה-process group id שלהם הוא של השולח.
  • pid == -1 — שולח לכל התהליכים חוץ מ-init.

הפונקציה pause

#include <unistd.h> int pause(void);

מכניסה את התהליך למצב "הפסקה" עד שמתקבל איתות מתהליך אחר. הפונקציה חוזרת (משתחררת) רק כאשר פונקציית handler תפסה איתות וגם סיימה את פעולתה. ערך החזרה הוא −1.

pause_question.c
int main(void) {
    printf("waiting for SIGCHLD\n");
    pause();
    printf("waiting for SIGCHLD again\n");
    pause();
    printf("I'm done\n");
    return 0;
}

שאלה: תהליך כלשהו שולח ברצף SIGCHLD (שפעולת ברירת המחדל שלו היא ignore) לתוכנית. מה יודפס?

פתרון: יודפס רק פעם אחת. אחרי ההדפסה הראשונה התהליך נכנס למצב "הפסקה" עד שיקבל איתות שהוגדר עבורו טיפול. אבל מכיוון שלא הוגדר טיפול ב-SIGCHLD — הוא לא יתעורר עקב הגעת איתות זה, ולכן יישאר תקוע ב-pause הראשון.

הפונקציה alarm

#include <unistd.h> unsigned int alarm(unsigned int seconds);

תהליך מבקש שישלחו אליו SIGALRM בעוד מספר שניות — שימושי להגבלת פעולה בזמן. כל קריאה ל-alarm מבטלת את הקודמת. הערך המוחזר: מספר השניות שחסרו עד שה-alarm הקודם היה אמור לקרות.

timeout.c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void on_alarm(int sig) {
    printf("Time's up!\n");
    _exit(1);
}

int main(void) {
    signal(SIGALRM, on_alarm);
    alarm(5);                 /* תזמון SIGALRM בעוד 5 שניות */
    printf("Enter input within 5 seconds:\n");
    char buf[100];
    fgets(buf, sizeof buf, stdin);   /* אם איטיים מדי — ה-alarm יקטע */
    return 0;
}

הסבר משולב: רושמים handler ל-SIGALRM, ואז alarm(5) קובע שבעוד 5 שניות יישלח האיתות. אם המשתמש לא מקליד בזמן — ה-handler ירוץ ויסיים את התוכנית. שילוב קלאסי של signal + alarm ליצירת timeout.

הפונקציה raise

#include <signal.h> int raise(int sig);

שליחת סיגנל ע"י התהליך לעצמו. sig הוא הסיגנל לשליחה. ערך חוזר: 0 בהצלחה, אחר בכישלון.

בקרת תהליכים · Foreground vs Background

הפקודה ps [options] נותנת צילום מצב של מה שרץ כרגע — ה-PID, הטרמינל (TTY), זמן המעבד, ושם התוכנית.

Foreground Job

התהליך ש"תופס" את הטרמינל. כל מה שמקלידים הולך אליו וכל מה שמדפיס מופיע על המסך. רק תהליך אחד יכול להיות ב-Foreground.

Background Job

תהליך שרץ "מאחורי הקלעים" — לא קשוב למקלדת, אבל ממשיך לפעול.

terminal
./a.out &       # דרך א': מריצים ישר ברקע, הטרמינל נשאר פנוי

# דרך ב': תוך כדי ריצה
# Ctrl+Z   → שולח SIGTSTP, התוכנית נכנסת למצב "קפוא" (Stopped)
bg              # ממשיך להריץ את התהליך האחרון שעצרת — אבל ברקע

למה זה שימושי? לפעולות ארוכות (קומפילציה גדולה, העתקת קבצים, הרצת שרת) — לא רוצים שהטרמינל יהיה "תפוס". & או bg משחררים אותו להמשך עבודה במקביל.

📖 תרגול 7 — המלך SIGKILL

בכל פעם שאתם לוחצים Ctrl+C אתם שולחים SIGINT (סיגנל 2) — בקשה ידידותית לעצור. אבל המלך האמיתי הוא SIGKILL (סיגנל 9): הסיגנל היחיד שהתהליך לא יכול לתפוס, לחסום או להתעלם ממנו. אין handler שיציל אתכם — הקרנל פשוט מוחק את התהליך מהזיכרון.

אבל יש מלכוד: אפילו SIGKILL יכול להיכשל. אם תהליך תקוע ב-Uninterruptible Sleep (מצב D ב-ps, בגלל המתנה לדרייבר או דיסק תקוע), הקרנל לא יכול להעיר אותו כדי להרוג אותו. התיעוד הרשמי של Linux אומר על זה: "זה מהווה באג במערכת ההפעלה שצריך לדווח עליו".

המסר: ארכיטקטורה טובה — בקוד, בצוות או בחיים — נמדדת ביכולת לעשות Graceful Shutdown לאיתותים קטנים, לפני שהמציאות שולחת SIGKILL.
