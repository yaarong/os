File Descriptor

בלינוקס "הכל הוא קובץ". כשמערכת ההפעלה רוצה לתת לתוכנית גישה — היא לא נותנת לגעת בחומרה ישירות, אלא מביאה "כרטיס": FD. לכל תהליך טבלה פרטית משלו שמנוהלת ע"י ה-OS; FD הוא מספר שלם חיובי שמייצג שורה בטבלה. בכל פתיחת תהליך נפתחים שלושה ערוצים ראשוניים (IN/OUT/ERR), ולכן קבצים נוספים יקבלו FD מ-3 ומעלה.

Open File Description

מבנה בזיכרון מערכת ההפעלה שמייצג מופע ספציפי של קובץ פתוח — לא את הקובץ עצמו שבדיסק, אלא את מצב העבודה הנוכחי מולו. שומר:

  • המיקום הנוכחי בקובץ (offset).
  • האם הקובץ לקריאה / כתיבה / קריאה וכתיבה.
  • דגלים והערות נוספות.
  • מצביע ל-inode עבור הקובץ.
📌 שיתוף offset

אם זוג FD מצביעים לאותו משאב בטבלת הקבצים הפתוחים (למשל אחרי dup2()) — הם חולקים את אותו file offset ואת אותם דגלי גישה.

קריאת המערכת open + הרשאות

int open(const char *pathname, int flags); int open(const char *pathname, int flags, mode_t mode);

הדגלים: O_RDONLY, O_WRONLY, O_RDWR, O_CREAT, O_EXCL, O_TRUNC, O_APPEND.

⚠️ אל תשכחו את mode

אם יוצרים קובץ עם O_CREAT בלי להעביר mode — התוכנית תתקמפל, אבל הקובץ ייווצר עם הרשאות אקראיות ממה שהיה בסטאק (עלולים לנעול את עצמכם מחוץ לקובץ).

קבוצהReadWriteExecuteשילוב מלא
User (Owner)S_IRUSR (400)S_IWUSR (200)S_IXUSR (100)S_IRWXU (700)
GroupS_IRGRP (040)S_IWGRP (020)S_IXGRP (010)S_IRWXG (070)
OthersS_IROTH (004)S_IWOTH (002)S_IXOTH (001)S_IRWXO (007)
create_644.c
/* קובץ שלבעליו מותר לקרוא ולכתוב, ולכל השאר רק לקרוא (0644) */
int fd = open("data.txt", O_CREAT | O_WRONLY,
              S_IRUSR | S_IWUSR | S_IRGRP | S_IROTH);

הפונקציה access

#include <unistd.h> int access(char* pathname, int mode);

בודקת אם לתהליך הקורא יש גישה לקובץ. mode:

  • R_OK — גישת קריאה?
  • W_OK — גישת כתיבה?
  • X_OK — גישת הרצה?
  • F_OK — האם הקובץ קיים?

יש גישה → 0, אחרת −1.

FHS & Inode

תקן ה-FHS (File Hierarchy Standard) מגדיר את מבנה התיקיות והתוכן שלהן בלינוקס. כל הקבצים מופיעים תחת השורש /, גם אם הם מאוחסנים בהתקן (פיזי/וירטואלי) שונה.

לכל File system יש טבלה של inodes (Index Node) — מבנה נתונים בסיסי שמכיל את כל המטה-דאטה על הקובץ, פרט לשם שלו ולתוכן הממשי: גודל הקובץ, מס' ה-USER וה-GROUP, רמת גישה ועוד.

מבנה התיקייה

תיקייה היא קובץ לכל דבר — ההבדל היחיד הוא שתוכנה הוא טבלה של מיפויים. כל שורה מייצגת קובץ/תת-קובץ, ומורכבת מ:

  1. שם הקובץ (מחרוזת טקסט).
  2. מספר ה-inode — מוביל ל-inode בטבלה הגלובלית.

זוג כזה נקרא directory entry (או link). כשמבקשים לגשת לקובץ, המערכת פותחת את קובץ התיקייה, מחפשת את השם, שולפת את מספר ה-inode, ניגשת לטבלה הגלובלית, ומשם שולפת את השדה שמצביע פיזית לדיסק.

terminal
ls -i    # מציג את מספר ה-inode של כל קובץ/תיקייה

Hard Link

מצב שבו יש יותר מקובץ אחד שמצביע לאותו מספר inode בדיוק. אין כאן "מקור" ו"העתק" — שני השמות שקולים לחלוטין, מובילים לאותו מבנה נתונים ולאותו מיקום פיזי בדיסק.

int link(const char *oldpath, const char *newpath);

מקבלת נתיב לקובץ קיים (שיש לו שם ו-inode) ושם חדש שיתווסף בטבלת התיקייה עבור אותו inode. 0 בהצלחה, −1 בכישלון.

⚠️ מגבלות Hard Link
  • אי אפשר ליצור בין מחיצות שונות.
  • אי אפשר ליצור לתיקייה — מערכת הקבצים בנויה כעץ; קישור קשיח לתיקייה עלול ליצור לולאה אינסופית (תיקייה שמכילה קישור לאב שלה).
terminal
echo A > file1.txt
ln file1.txt file2.txt   # יוצר hard link — מונה הקישורים ל-inode = 2

rm file1.txt             # המונה יורד מ-2 ל-1 → הקובץ עדיין חי!
cat file2.txt            # → A   (התוכן נגיש דרך השם השני)

הסבר: rm מבצע unlink — מסיר שורה מהתיקייה ומפחית את מונה הקישורים. הקובץ נמחק פיזית רק כשהמונה מגיע ל-0.

Symbolic (soft) Link

בניגוד ל-Hard Link — קובץ חדש ונפרד לחלוטין, עם inode משלו ובלוקים משלו. בתוכו מחרוזת טקסט שמכילה את הנתיב לקובץ המקורי.

int symlink(const char *oldpath, const char *newpath);

היתרונות על Hard Link: עובד בין מחיצות שונות (אין מצביע ל-inode פנימי, רק טקסט נתיב), ומותר ליצור לתיקיות.

terminal
touch file1.txt
ln -s file1.txt file2.txt   # soft link → file2.txt -> file1.txt
readlink file2.txt          # מציג את הנתיב ששמור בתוך הקישור

איך הקרנל מתנהג כשפותחים soft link?

  1. ב-open() על קישור סימבולי — המערכת פותחת את ה-inode של הקישור, רואה שתוכנו נתיב, ומפנה את עצמה אוטומטית לנתיב.
  2. פתרון לולאות: אם קישור מצביע על קישור אחר וכו' — המערכת עוקבת עד שתגיע לקובץ אמיתי. כדי למנוע לולאה אינסופית יש מגבלה של כ-40 מעברים, ואם עברנו אותה — open ייכשל.
  3. בהגעה לקובץ היעד — שולפת את ה-inode האמיתי, פותחת ומחזירה fd.

Broken / Dangling Link

ה-Soft Link לא מכיר את מספר ה-inode של היעד — רק מחרוזת נתיב. אם הקובץ המקורי נמחק או זז — הקישור עדיין קיים, אך ניסיון לגשת אליו ייכשל (הנתיב לא נמצא). זה קישור שבור — הטרמינל מסמן אותו באדום.

Stat

#include <sys/stat.h> int stat(const char* path, struct stat* buf); int lstat(const char* path, struct stat* buf); // עבור soft link עצמו ולא היעד int fstat(int fd, struct stat* buf); // עבור file descriptor

משפחת פונקציות ששולפת את ה-inode של הקובץ מהדיסק וממלאת מבנה נתונים. ערך מוחזר: 0 בהצלחה, −1 בכישלון.

stat_demo.c
#include <stdio.h>
#include <sys/stat.h>

int main(int argc, char **argv) {
    struct stat sb;
    if (stat(argv[1], &sb) == -1) {
        perror("stat");
        return 1;
    }
    printf("inode:  %lu\n", (unsigned long)sb.st_ino);
    printf("size:   %ld bytes\n", (long)sb.st_size);
    printf("links:  %lu\n", (unsigned long)sb.st_nlink);
    return 0;
}

הקלט: argv[1] — נתיב לקובץ. הפלט: מספר ה-inode, גודל הקובץ ומספר הקישורים הקשיחים אליו — כל אלו נשלפים מתוך ה-inode דרך struct stat.

📖 תרגול 9 — Git, Hard Links ו-stat

הפקודה git clone --local פועלת מיידית כי Git אינו מעתיק את קבצי האובייקטים, אלא יוצר עבורם Hard Links — שתי רשומות ספרייה המצביעות לאותו inode. באופן דומה, היעילות של git status נובעת מ-stat(), שמאפשרת ל-Git להשוות רק את המטא-דאטה (גודל וזמני שינוי) מול האינדקס, בלי לקרוא את תוכן הקבצים.

מאחר שניהול ה-inodes מוגבל למערכת הקבצים המקומית, Hard Links אינם חוצים מחיצות — ולכן mv מיידי באותה מחיצה (שינוי רשומת ספרייה), אך הופך ל-Copy+Delete איטי בין מערכות קבצים שונות.

המסר: בחיים תצטרכו גם Soft Links — קשרים גמישים שמאפשרים לדלג בין עולמות; וגם Hard Links — חיבורים עמוקים ויציבים לשורש הדבר, שלא יישברו גם אם כל התפאורה מסביב תשתנה.
