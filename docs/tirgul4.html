חוטים — Threads

חוט הוא יחידת ביצוע עצמאית שפועלת בתוך תהליך ("שורת עבודה"). בלינוקס תהליך אחד יכול לכלול מספר חוטים, שחולקים ביניהם את משאבי התהליך — בין היתר code, data, heap ומשאבי מערכת.

למרות השיתוף, כל חוט הוא "הקשר ביצוע" נפרד. לכל חוט באופן בלעדי:

  1. מחסנית — לניהול קריאות לפונקציות ומשתנים מקומיים.
  2. Program Counter — מצביע על הפקודה הבאה שהחוט אמור לבצע.
  3. סט רגיסטרים — המצב הנוכחי של המעבד עבור אותו חוט.

יתרונות החוטים

  1. ביצוע חלקים בלתי-תלויים במשימה — הרצה בו-זמנית של חלקים שונים שאינם תלויים זה בזה.
  2. מקביליות על מעבדים שונים — הרצת מספר חוטים במקביל על מעבדים שונים, מה שמזרז את זמן הביצוע הכולל.
  3. שיפור ביצועים גם במעבד יחיד — בזמן שחוט אחד מבצע פעולה חוסמת (המתנה ל-IO מהדיסק), חוט אחר ממשיך לעבוד במקום שהמעבד יתבזבז על המתנה.
💡 חוט זול מתהליך

הוספת חוט זולה בהרבה מהוספת תהליך שלם — היא לא דורשת הקצאת זיכרון חדש או הרשאות חומרה, שכן הם משותפים לכל החוטים. עם זאת — השיתוף מחייב לתאם בין החוטים שניגשים לאותם משתנים, כדי למנוע שיבוש נתונים.

Process Descriptor

בלינוקס, לכל תהליך קיים בקרנל מבנה נתונים שנקרא מתאר תהליך (process descriptor) — רשומה שמרכזת את כל המידע על התהליך:

  1. מצב התהליך (רץ / ממתין / סיים)
  2. עדיפות התהליך (כמה זמן מעבד יקבל ביחס לאחרים)
  3. מזהה התהליך — PID
  4. מצביע לטבלת אזורי הזיכרון
  5. מצביע לטבלת הקבצים הפתוחים

POSIX Threads & Thread Group

התקן POSIX Threads דורש שכל החוטים של אותו תהליך יהיו מאוגדים יחד — קריאה ל-getpid() מכל חוט אמורה להחזיר את אותו ערך.

⚠️ הקאץ' של לינוקס

בלינוקס היחס לכל חוט הוא כאל תהליך רגיל — לכל חוט מתאר משלו ובפרט PID משלו. אך התקן מצפה ש-PID יזהה את כל החוטים. לכן לינוקס משתמשת ב-Thread Group.

איך לינוקס מממשת זאת:

  • Thread_group — מתארי כל החוטים מקושרים פיזית למתאר ה-primary thread (החוט הראשון).
  • Tgid (thread group id) — ה"שקר המוסכם": מכיל את ה-PID המשותף לכל החוטים בקבוצה. בפועל זהו ה-PID של החוט הראשון.
Thread AThread BThread C
PPID290029002900
TGID (←getpid)300030003000
TID300030013002

לכן בלינוקס getpid() מחזירה את ה-TGID של החוט הקורא. פעולות על ה-PID המשותף מתורגמות לפעולה על קבוצת החוטים המתאימה.

ה-API של pthread

תחילה יש לבצע #include <pthread.h> ולקשר עם -pthread.

pthread_create — יצירת חוט

int pthread_create( pthread_t *thread, // יישמר מזהה החוט החדש const pthread_attr_t *attr, // מאפיינים (לרוב NULL) void *(*start_func)(void*), // הפונקציה שהחוט יריץ void *arg); // ארגומנט לפונקציה

יוצרת חוט חדש שרץ במקביל לחוט הקורא, בתוך אותו תהליך. פונקציה אסינכרונית — החוט המקורי ממשיך לשורה הבאה מבלי לחכות. מחזירה 0 בהצלחה, או קוד שגיאה חיובי בכישלון. דומה ל-fork, רק ש-fork משכפל תהליך — וכאן מוסיפים "פועל" ידנית לתהליך.

pthread_self · join · cancel · exit

pthread_t pthread_self(void); // מזהה החוט הקורא int pthread_join(pthread_t th, void **thread_return); // כמו wait לחוט int pthread_cancel(pthread_t thread); // בקשת ביטול void pthread_exit(void *retval); // סיום עצמי של החוט
  • pthread_join — החוט הקורא עוצר וממתין עד שהחוט th מסיים. ההמתנה משחררת את מידע הניהול של החוט; בלעדיה נשארות "שאריות" של חוטים מתים. ניתן להמתין לחוט מסוים פעם אחת בלבד, וכל חוט יכול להמתין לכל חוט אחר.
  • pthread_cancel — שולחת בקשה לסיים חוט (לא פקודה חד-משמעית). החוט לא תמיד מת מיד — תלוי בנקודת ובמצב הביטול שהוגדר.
  • pthread_exit — החוט מחליט על דעת עצמו לסיים. retval מצביע לערך שמי שיבצע join יקבל. אם ה-main קורא לה — הוא מסיים את עצמו אך לא סוגר את התהליך; שאר החוטים ימשיכו עד שהאחרון יסיים.

דוגמה משולבת: יצירת חוטים והמתנה

threads_demo.c
#include <stdio.h>
#include <pthread.h>

void *worker(void *arg) {
    int id = *(int *)arg;
    printf("Thread %d running\n", id);
    return NULL;          /* ערך זה יוחזר למי שיעשה join */
}

int main(void) {
    pthread_t t[3];
    int ids[3] = {0, 1, 2};

    for (int i = 0; i < 3; i++)
        pthread_create(&t[i], NULL, worker, &ids[i]);

    for (int i = 0; i < 3; i++)
        pthread_join(t[i], NULL);   /* ממתינים שכל חוט יסיים */

    printf("All threads done\n");
    return 0;
}

הסבר משולב: ה-main יוצר 3 חוטים, וכל אחד מקבל מצביע ל-id שלו כ-קלט (arg). כל חוט מריץ את worker ומדפיס את ה-id. לולאת ה-join מבטיחה שה-main ימתין לכל החוטים לפני שהוא ממשיך — בלעדיה ה-main היה עלול לסיים את התהליך לפני שהחוטים הספיקו לרוץ.

סיום חוט — מתי?

  1. חזרה מהפונקציה הראשית של החוט.
  2. קריאה ל-pthread_exit() בתוך החוט.
  3. קריאה ל-exit() ע"י חוט כלשהו בקבוצה — מסיימת את כל התהליך.
  4. סיום טבעי של החוט הראשי.
  5. הריגת החוט ע"י pthread_cancel מחוט אחר (או מעצמו).

fork & exec בתוך חוט

fork() בתוך חוט

נוצר תהליך חדש שהוא בן של החוט הקורא. רק חוט אחד משתכפל לתהליך הבן (גם אם לתהליך המקורי היו עוד חוטים). חוט אחר בקבוצה יכול לבצע wait על תהליך הבן.

exec() בתוך חוט

תהליך דרסטי: ברגע שהקריאה מצליחה, המערכת מוחקת את כל שאר החוטים בתהליך מיד (ללא join). החוט שביצע את הקריאה הוא הניצול היחיד, והופך לחוט הראשי של התוכנית החדשה. כל מרחב הכתובות נמחק ונוצר מחדש.

Mutex

מנגנון סנכרון שמונע מכמה חוטים לגשת למשאב משותף בו-זמנית. עובד כמו מפתח יחיד לחדר: מי שמחזיק במפתח נכנס, וכל השאר מחכים בחוץ עד שהוא יצא ויחזיר את המפתח.

  • משתנה פשוט שיכול להיות נעול או לא נעול.
  • מקיף קוד רגיש שניגש לנתונים משותפים — נועלים בכניסה, משחררים ביציאה.
  • לכל היותר חוט אחד מחזיק בו; חוט שינסה לנעול מנעול תפוס — נחסם.
  • בעלות: רק החוט שנעל את המיוטקס מורשה לשחרר אותו.

אתחול וסוגי mutex

int pthread_mutex_init(pthread_mutex_t *mutex, const pthread_mutexattr_t *attr);
סוגתיאור
PTHREAD_MUTEX_NORMALרגיל ומהיר — לא בודק טעויות (נעילה כפולה ע"י אותו חוט → DEADLOCK).
PTHREAD_MUTEX_RECURSIVEרקורסיבי — מאפשר לאותו חוט לנעול מספר פעמים ברצף בלי להיחסם.
PTHREAD_MUTEX_ERRORCHECKבודק שגיאות ומחזיר קוד שגיאה על פעולה לא חוקית (שחרור מנעול של אחר).
PTHREAD_MUTEX_DEFAULTברירת המחדל של המערכת (בלינוקס — מתנהג כמו NORMAL).
⚠️ אסור לאתחל פעמיים

אתחול מיוטקס שכבר אותחל → התנהגות לא מוגדרת. גם הריסת מנעול נעול/לא-מאותחל → התנהגות לא מוגדרת.

פעולות על mutex

int pthread_mutex_lock(pthread_mutex_t *mutex); // נעילה (חוסם עד שמתפנה) int pthread_mutex_trylock(pthread_mutex_t *mutex); // ניסיון נעילה (לא חוסם) int pthread_mutex_unlock(pthread_mutex_t *mutex); // שחרור int pthread_mutex_destroy(pthread_mutex_t *mutex); // הריסה

דוגמה: הגנה על מונה משותף

mutex_counter.c
#include <stdio.h>
#include <pthread.h>

#define N 1000000
long counter = 0;
pthread_mutex_t lock;

void *inc(void *arg) {
    for (int i = 0; i < N; i++) {
        pthread_mutex_lock(&lock);     /* כניסה לקטע קריטי */
        counter++;                      /* משאב משותף */
        pthread_mutex_unlock(&lock);   /* יציאה */
    }
    return NULL;
}

int main(void) {
    pthread_t a, b;
    pthread_mutex_init(&lock, NULL);

    pthread_create(&a, NULL, inc, NULL);
    pthread_create(&b, NULL, inc, NULL);
    pthread_join(a, NULL);
    pthread_join(b, NULL);

    pthread_mutex_destroy(&lock);
    printf("counter = %ld\n", counter);  /* תמיד 2000000 */
    return 0;
}

הסבר משולב: שני חוטים מגדילים את counter מיליון פעמים כל אחד. ללא המיוטקס, counter++ (שהוא קריאה→הגדלה→כתיבה) היה נתון ל-race condition, והתוצאה הסופית הייתה נמוכה מ-2,000,000. עם lock/unlock — רק חוט אחד נמצא בקטע הקריטי בכל רגע, ולכן התוצאה תמיד נכונה.

🎁 סיפור סוף התרגול

📖 תרגול 4 — Therac-25

בין 1985 ל-1987, מכונת ההקרנות Therac-25 העבירה מנות קרינה מסיביות לפחות ל-6 מטופלים — חלקם קיבלו מאות מונים מהמינון המתוכנן. שלושה מטופלים מתו. הסיבה? Race Condition. לתוכנה היו שגרות מקבילות שחלקו משתנה בלי סנכרון נכון (בלי mutex!).

אם מפעיל הקליד מהר מספיק במהלך חלון של 8 שניות, התוכנה נכנסה למצב לא עקבי — הפעילה את הקרן בעוצמה מלאה בלי מפזר הבטיחות. אותו באג התקיים בדגם הקודם (Therac-20), אבל נעילות חומרה הסתירו אותו. כשהוחלפו בבדיקות תוכנה בלבד, ה-race condition הפך קטלני.

המסר: שלושה אנשים מתו בגלל mutex חסר. המושגים שלמדתם הם לא אקדמיים — הם מצילים חיים.
