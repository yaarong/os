הזיכרון הפיזי

זיכרון שנמצא בהתקנים הפיזיים של המחשב:

RAM — ראשי

מאחסן את התוכנית לביצוע והנתונים שהיא משתמשת בהם. זיכרון נדיף.

דיסק קשיח — משני

מאחסן כמות גדולה מאוד של נתונים. זיכרון שאינו נדיף.

הזיכרון הווירטואלי

מרחב זיכרון מדומה שעומד לרשות תהליך — ויכול להיות גדול יותר מהזיכרון הפיזי הראשי. הוא ממופה בחלקו ל-RAM ובחלקו לזיכרון המשני. הקוד שמבוצע והנתונים הדרושים חייבים להיות ב-RAM בזמן הביצוע.

💡 אנלוגיה — שולחן וארון ספרים

ה-RAM הוא שולחן עם ספרים; הזיכרון המשני הוא ארון ספרים ענק. הזיכרון הווירטואלי מאפשר לעבוד כאילו השולחן ענק — גם כשהוא פיזית קטן:

  1. המחשב אומר לכל תוכנה: "קחי מרחב זיכרון עצום, תעשי בו מה שתרצי". התוכנה לא יודעת שזה לא אמיתי.
  2. המיפוי: רק הדפים (PAGES) שהתוכנה באמת צריכה כעת מונחים על ה-RAM.
  3. אם תוכנה צריכה מידע שבארון — המחשב מפנה מקום בשולחן (מחזיר לזיכרון המשני דף שלא בשימוש).

למה צריך זיכרון וירטואלי?

  1. הפרדה בין תהליכים: לכל תהליך מרחב וירטואלי משלו. המערכת מפרידה את מרחב הכתובות של כל תהליך — ולכן תהליך אחד לא יכול לפגוע בנתונים של אחר.
  2. הגדלת הזיכרון: בכל רגע רק חלק ממרחב הזיכרון של תהליך נמצא ב-RAM. מאפשר להריץ תהליכים שסך הזיכרון שלהם גדול מה-RAM (או תהליך בודד שמרחבו גדול מה-RAM).

דפים ומסגרות

  • לכל תהליך מרחב זיכרון וירטואלי משלו, מתחיל לרוב בכתובת 0. ניתן לייצג עד 256TiB−1 כתובות.
  • המרחב מחולק לדפים (pages) — קטעי זיכרון עוקבים בגודל קבוע (בד"כ 4KB, תלוי מערכת).
  • ה-RAM מחולק למסגרות (frames), כל אחת בגודל דף.
  • דף שנמצא ב-RAM מקושר למסגרת כלשהי.

Shared Memory

נזכר ב-PIPES וב-FIFO: בכל קריאה/כתיבה יש context-switch למערכת ההפעלה. במקום זאת, ניתן להגדיר אזור זיכרון משותף לכמה תהליכים — IPC Shared Memory. בפועל: ממפים את אותה המסגרת (frame) למספר דפים של תהליכים שונים.

יתרונות:

  1. מאפשר למספר תהליכים לגשת לאותו מקום בזיכרון ולחלוק מידע.
  2. בניגוד ל-pipe — קריאה לא מרוקנת את הנתונים (כמו זיכרון רגיל).
  3. כל התהליכים רואים את המידע שהשתנה.
  4. מהיר בהרבה מ-pipes.
  5. לא דורש קריאות מערכת בכל כתיבה/קריאה.

יצירת זיכרון משותף — שלבים

1. מפתח ייחודי — ftok

key_t ftok(char *pathname, char proj_id);

הפרמטרים: שם קובץ קיים, ותו כלשהו שיזוהה עם המפתח.

2. הקצאה — shmget

#include <sys/ipc.h> #include <sys/shm.h> int shmget(key_t key, size_t size, int shmflg);

מקצה אזור משותף (במפתח שאף אחד לא נתן עדיין) או מקבל מזהה של אזור קיים. key — מפתח (או IPC_PRIVATE למקום חדש), size — גודל בבתים, shmflg — דגלים והרשאות. מחזיר shmid (מזהה ייחודי) או −1.

3. התחברות — shmat

#include <sys/types.h> #include <sys/shm.h> void *shmat(int shmid, const void *shmaddr, int shmflg);

מקשרת את הזיכרון שהוקצה לתהליך (זה פיזית החץ בין טבלת הדפים לפריים). shmaddr — לרוב NULL. מחזירה כתובת במרחב הזיכרון של התהליך, או −1.

4. ניתוק — shmdt

int shmdt(const void *shmaddr);

מקבלת את הכתובת אליה ממופה הזיכרון המשותף, ומנתקת את הקשר בין התהליך לזיכרון.

דוגמה מלאה

shared_mem.c
#include <stdio.h>
#include <string.h>
#include <sys/ipc.h>
#include <sys/shm.h>

int main(void) {
    /* 1) מפתח ייחודי מתוך קובץ קיים + מזהה פרויקט */
    key_t key = ftok("/tmp", 'A');

    /* 2) הקצאת 1024 בתים של זיכרון משותף (יצירה אם לא קיים) */
    int shmid = shmget(key, 1024, IPC_CREAT | 0666);

    /* 3) התחברות — קבלת מצביע לזיכרון המשותף */
    char *shm = (char *)shmat(shmid, NULL, 0);

    /* כתיבה — כמו לזיכרון רגיל, בלי קריאת מערכת */
    strcpy(shm, "Hello shared world!");
    printf("Wrote: %s\n", shm);

    /* 4) ניתוק בסיום השימוש */
    shmdt(shm);

    /* 5) מחיקת המקטע מהמערכת (אחרת יישאר גם אחרי סיום) */
    shmctl(shmid, IPC_RMID, NULL);
    return 0;
}

הסבר משולב: ftok מייצר מפתח (קלט: נתיב /tmp ותו 'A') ששני תהליכים נפרדים יכולים לחשב בצורה זהה כדי לקבל את אותו אזור. shmget מקצה 1024 בתים ומחזיר shmid. shmat ממפה את האזור לכתובת בתהליך, ומאותו רגע כותבים/קוראים כמו זיכרון רגיל — ללא קריאות מערכת, ומכאן המהירות. בסיום מנתקים עם shmdt ומוחקים את המקטע עם shmctl(IPC_RMID).

shmctl — מחיקה/עדכון

#include <sys/shm.h> int shmctl(int shmid, int cmd, struct shmid_ds *buf);

מאפשר פעולות על ה-shared memory: שליפה ועדכון של פרמטרים, ובקשה להשמדת השטח. פרמטרים:

  • shmid — מזהה האובייקט (מ-shmget).
  • cmd — הפקודה:
    • IPC_STAT — שליפת הסטטוס של ה-shared memory לתוך buf.
    • IPC_SET — שינוי שדות מסוימים (uid/gid/mode) לפי buf.
    • IPC_RMID — מחיקת המקטע. רק owner/creator או root יכולים.
  • buf — מבנה הנתונים אליו יוקרא / ממנו יילקח הסטטוס.

ערך מוחזר: 0 בהצלחה, −1 בכישלון.

📖 תרגול 11 — Chrome ו-Shared Memory

פתחו Chrome וגשו ל-chrome://system — תראו עשרות תהליכים. כל טאב רץ בתהליך נפרד (לביטחון ובידוד קריסות), אבל הם צריכים לחלוק נתונים. Chrome משתמש ב-shared memory בצורה נרחבת: תהליך ה-GPU חולק פריימים מרונדרים עם תהליכי הטאבים דרך buffers של shared memory, ונמנע מפעולות copy יקרות.

כשטאב אחד קורס, האחרים שורדים כי הם תהליכים נפרדים — אבל הם עדיין יכולים לתקשר ביעילות, כי shared memory לא דורש מעורבות קרנל בכל קריאה/כתיבה.

המסר: כמו בקוד, גם בחיים צריך לפעמים לבנות ערוצי תקשורת קבועים ומרחב ידע משותף ללא מתווכים. זה הסוד שמונע מהמערכות הכי מורכבות — ומאיתנו — לקרוס ולבזבז זמן מיותר.
